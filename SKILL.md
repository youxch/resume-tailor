# Resume Tailor — 简历定制 Skill 模板

你在帮 **[候选人姓名]** 针对特定岗位定制简历。

> 这是一个可复用的 skill 模板。下面出现的"方向A / 方向B"是一个示例（来自电磁工程 + AI 算法的交叉背景），**实际使用前请把它替换成你自己的两个专业方向**，并在 references 模板文件里填入你自己的真实背景。

**候选人背景放在两个 references 文件中（按模板自行填写）：**
- `references/candidate-profile.template.md` — 教育、技能、论文、荣誉、开源影响力
- `references/project-details.template.md` — 核心项目的详细内容和不同方向下的包装方案

需要时读取这些文件。对于求职者熟悉的常见公司，可以提前了解足够背景，减少临时搜索。

## 工作流程

### 第一步：获取JD信息

如果用户只给了公司+岗位名（没贴JD），先问他："能把岗位JD贴给我吗？没有的话告诉我岗位名称和公司，我来判断方向。"

如果用户已给JD，直接进入分析。

### 第二步：判断方向

读取JD，判断岗位属于哪个方向：

**方向A — 天线/RF/电磁工程**
关键词：天线设计、射频、电磁仿真、超构表面、MIMO、波束赋形、天线测试、CMA、HFSS/CST

**方向B — AI算法/大模型**
关键词：深度学习、机器学习、生成模型、大模型、AI4S、仿真加速、逆向设计、科学计算、数据驱动

**混合型** — JD同时包含两方向（如Ansys、CST/达索系统、仿真软件AI化岗位）：选取两者最强的交叉点

判断后明确告诉用户："这个岗位是**方向X**，我将用对应策略定制。"

### 第三步：选定内容策略

读取 `references/project-details.template.md`，根据方向选择项目和论文的排序与侧重。下面用占位符演示这套"按方向重排素材"的逻辑——把 [项目X]/[论文X] 换成你自己的真实内容：

**方向A策略示例：**
- 项目排序：[项目1]（突出与方向A强相关的方法+工程交付）→ [项目2]（突出可迁移到方向A的子能力）→ [项目3]（作为背景补充）
- 论文：[最贴合方向A的论文] → [次贴合的论文]
- 强调：与方向A关键词强相关的方法论、物理验证/工程交付、对应领域的专业工具
- 技能关键词：从JD和你的真实技能交集中提取3-6个

**方向B策略示例：**
- 项目排序：[项目3]（突出与方向B强相关的算法/架构）→ [项目1]（突出可迁移的数据/算力规模）→ [项目2]
- 论文：[最贴合方向B的论文] → [开源/工具型产出] → [其余论文作为背景]
- 强调：核心算法/架构、数据规模、量化加速比、开源影响力
- 技能关键词：从JD和你的真实技能交集中提取3-6个

**混合型策略示例：** 找到"用方向B的工具做方向A的事"这类交叉定位，开头用一句话点明——*"长期深耕[方向A领域]，同时开发了基于[方向B方法]的加速/分析框架，提速 [数字]x。"*

### 第四步：生成定制简历内容

**先读 `references/style-guide.md`，然后按以下结构输出。**

节顺序（默认，可按方向微调）：
```
求职意向（一行，按方向写）
教育背景
科研项目（核心，最长最详细）
专业技能
个人荣誉
发表论文
```

**每个科研项目必须用 4 段式结构（严格按此模板，不得改动结构或段落名称）：**

```
[时间段] ………… [项目名称]（[来源]）……… 主要负责人

1. 研究目标与意义：针对[行业/工程痛点]，[本项目做了什么]，[意义/价值]。
   （1-2行，目标+意义合并，不拆开）

2. 研究内容：①[方法名]；②[方法名]；③[方法名]；④[方法名]。
   （全部写在一行，分号分隔，紧凑不换行）

3. 关键技术进展与核心技术创新：
   ● 新[XX]：[背景/区别于传统]，首次[做了什么]，[结果**数字加粗**]
   ● 新[XX]：[背景/区别于传统]，首次[做了什么]，[结果**数字加粗**]
   ● 新[XX]：首次实现[成果]，[结果**数字加粗**]

4. 研究成果：[1行，论文+奖项+开源影响力]
```

**格式强制规范（违反即返工）：**

**参考模板可吸取的优点（不是照抄，是提取原则）**：
1. **"新XX"子标签**：新原理/新方法/新架构/新性能等，2-4字，HR扫一眼就知道"这人有创新点"——这是最值得吸取的设计
2. **研究内容一行分号格式**：`①XX；②XX；③XX；④XX。` 节省1-2行，内容不重要，标签即可
3. **每项目3个bullet**：强迫只保留最核心3个，超出就删或合并

**不要照抄的部分（保持用户自己的模板结构）**：
- 段落名称以用户模板为准（用户模板写"关键结果"就用"关键结果"，不擅自改为"关键技术进展与核心技术创新"）
- 研究目标单独一段（不强制合并"目标与意义"）
- 整体版式、字体、间距以用户自己的 DOCX 模板为准

**段落名称**（一字不差，照抄）：
- 第1段：`研究目标与意义：`
- 第2段：`研究内容：`
- 第3段：`关键技术进展与核心技术创新：`
- 第4段：`研究成果：`

**研究内容格式**：
- 全部写在1行，分号分隔：`①XX；②XX；③XX；④XX。`
- 不展开，不加括号说明

**关键技术子标签（"新XX"格式，2-4字）**：
- 用 `新原理` / `新方法` / `新架构` / `新算法` / `新工具` / `新性能` / `新验证` / `新数据` 等
- **禁止**用过程词（数据准备/网络训练）或功能词（并行仿真基础设施）
- 每个项目**3个**子标签，最多4个

**加粗规范**：
- 加粗放在**句中/句末的关键数字和核心创新短语**上
- 禁止在句首用加粗当分类标签

**行数控制（最重要，超出即压缩，生成后必须逐段数行）**：

每行约30个中文字（Word宋体10.5pt，A4页面正常页边距）。计算方法：内容字数 ÷ 30 = 行数，向上取整。

| 段落 | 最大行数 | 超出时的压缩方法 |
|------|---------|---------------|
| 项目标题行 | 1行 | 不可压缩 |
| 研究目标与意义 | 2行（≤60字） | 删背景铺垫，只留核心问题+目标 |
| 研究内容 | 1行（≤30字） | 4个词，分号分隔，绝不换行 |
| 关键技术（每个bullet） | 2行（≤60字） | 删修饰词，保留"首次+方法+数字" |
| 关键技术（3个bullet合计） | 6行 | 超出时删最弱一条，或将两条合并 |
| 研究成果 | 1行（≤40字） | 只保最强论文+最强奖项+开源影响力 |
| **单项目合计** | **≤11行** | |

两项目同时在简历中时，总行数 ≤ 22行，为其他模块（教育/技能/论文/荣誉）留足空间。

**其他规范**：
- 每个bullet结构：`[背景/区别于传统] + 首次[做了什么] + [量化结果**加粗**]`
- "首次"是核心词，每个创新点前必须有"首次"或"区别于X首次"
- 全文无"我"字，动词直接开头：提出/构建/搭建/实现/突破/制备/验证
- 涉及保密/敏感合作方时统一用"某重点单位合作项目"等模糊化表述，不要写出真实合作单位名称
- 论文后括注格式：（中科院1区TOP，第一作者，唯一学生作者）

**专业技能写宣言式段落，不是简单列表：**

方向A时：突出 精通电磁仿真（HFSS/CST）+ 精通天线测试 + 数理基础
方向B时：突出 精通AI4S研究范式（深度学习+高性能计算）+ 开源影响力 + 数理基础

### 第五步：公司定制备注

在简历内容之后，额外输出一段"**针对[公司名]的定制要点**"，说明：
1. 这份简历特别强调了什么（对应JD的哪些关键词）
2. 面试准备建议（1-2条）：比如"他们做HFSS，你可以提自己是深度用户"
3. 如果有遗漏数据（如FDA论文实验数据尚未填写），标记出来："注意：FDA论文草稿有12处待填实验数据，建议投递前更新"

## 受众视角闭环评审系统（自评→自改→多轮迭代，≥95分才输出）

### 核心原则

**这是一个自我驱动的闭环**：生成内容后，自己按三个受众视角打分 → 找到最低分角度 → 去指定位置找素材 → 按失分模式改法重写 → 再打分 → 循环直到≥95分。不需要用户介入，直接输出最终版。

---

### 第一步：受众画像（开始写之前，先确定这家公司谁会看）

| 受众 | 关注点 | 阅读时长 | 放下继续看的触发 |
|------|--------|---------|----------------|
| **HR/招聘** | JD关键词命中、数字醒目、没有废话 | 10秒 | 公司自己JD里的词、明确数字、知名奖项 |
| **技术面试官** | 方法可信、创新点清晰、能迁移、有工程验证 | 2分钟 | "首次"、背景→做了什么→量化结果三段结构、跨场景迁移 |
| **产品/业务负责人** | 能解决我们的问题吗、有没有落地 | 30秒 | 直接对应JD痛点、已交付/已开源/已验证 |

---

### 第二步：逐角度打分标准（满分100）

#### HR角度（权重30%）

| 评分项 | 分值 | 检查方法 |
|--------|------|---------|
| JD关键词镜像回写 | 30 | 逐条对照JD，每命中1个关键词+3分 |
| 关键数字10秒可扫出 | 30 | 加粗数字在句末，全文至少5个加粗数字 |
| bullet标题对HR有意义 | 40 | 标题是成果词/功能词，不是技术术语；军工/合同指标=直接-20 |

#### 技术面试官角度（权重40%）

| 评分项 | 分值 | 检查方法 |
|--------|------|---------|
| 每个bullet有三段结构 | 35 | 背景痛点→首次做了什么→量化结果，缺任一段-10 |
| 方法创新边界清晰 | 30 | "首次"后面跟具体条件限定（不只说"首次构建"，要说"在无需X的前提下首次提出Y"）|
| 工程完整性 | 20 | 有算法也有验证（开源/实测/第三方），纯算法无验证-15 |
| 可迁移性 | 15 | 至少1处说明"方法可迁移至X场景"，没有-15 |

#### 产品负责人角度（权重30%）

| 评分项 | 分值 | 检查方法 |
|--------|------|---------|
| 直接对应JD条目 | 40 | 每个项目至少1处括注"对应JD第X条"，或内容直接呼应 |
| 工具/成果可感知 | 35 | 你的代表性开源项目要说清楚用法（不只是GitHub浏览量），精度超商用软件说清楚"可集成为模块" |
| 无学术自嗨 | 25 | 成果描述有产品价值（节省X时间/降低Y成本/提升Z效率），不只说"发表论文X篇" |

---

### 第三步：失分模式 → 素材位置 → 改法（找到失分原因后按此表操作）

| 失分模式 | 素材在哪找 | 改法 |
|---------|----------|------|
| bullet标题是过程词（数据准备/网络训练）或受众不懂的术语 | 无需找素材，重新起名 | 改成**成果词**：并行仿真基础设施、逆向推理引擎、LLM仿真智能体 |
| 军工合同指标（新增模式2个/RCS+193%）写给软件公司 | 无需找新素材 | 翻译成工程价值：填补AI+仿真数据空白、精度超商用软件20倍 |
| 加粗在句首当标签，不在句末数字 | 无需找素材 | 把加粗移到句末关键数字上，句首标题改成普通文字 |
| "首次"边界太空（"首次构建平台"）| `references/project-details.template.md`（替换成你自己填好的版本） | 加条件限定："在无需物理先验知识的前提下，首次提出仅用统计分布特性选样的方法" |
| 缺具体数字/量化结果 | `references/project-details.template.md`（替换成你自己填好的版本） 对应项目的指标表 | 从文件中找到具体数字填入 |
| 开源项目描述太虚（只说浏览量）| `references/candidate-profile.template.md`（替换成你自己填好的版本） 开源影响力部分 | 说清楚工具流程：用户描述需求→参数配置→仿真调用→结果输出 |
| 缺论文/荣誉细节 | `references/candidate-profile.template.md`（替换成你自己填好的版本） 论文完整列表 | 按论文事实核查表核对状态再填入 |
| 不知道目标公司语言/战略 | WebSearch "[公司名] AI+仿真 战略 2025" + "[公司名] 招聘 研发方向" | 用公司自己说的词镜像回写到简历 |

---

### 第四步：迭代规则（自动执行，不需要用户催）

```
生成初稿
  ↓
按三角度打分（HR/技术/产品）
  ↓
综合加权分 ≥ 95？
  ├── 是 → 输出最终版 + 附三角度分数说明
  └── 否 → 找最低分角度 → 查失分模式表 → 去对应素材位置找资料 → 重写对应段落
              ↓
           重新打分 → 循环，直到 ≥ 95
```

输出时必须附：
```
HR角度：XX分 | 技术面试官：XX分 | 产品负责人：XX分
加权综合：XX分
主要改进：[第N轮改了什么/为什么]
```

## GitHub Pages 部署子流程（可选）

**触发词：** 上传github / 更新网页简历 / 部署到github / 让我能访问网页 / 更新index

把网页版简历部署成可公开访问的页面，思路很简单：把目标 HTML 文件放到你自己仓库根目录的 `index.html`，GitHub Pages 会自动识别为首页。

### 执行步骤

**第一步：把目标网页版简历复制成仓库根目录的 `index.html`**

```bash
cd "<你的本地仓库路径>"
python -c "import shutil; shutil.copy('简历/[公司]_[方向]_网页版.html', 'index.html')"
```

**第二步：提交并推送到你自己的远端仓库**

```bash
git add index.html
git commit -m "update: 首页换成[公司]_[方向]版本"
git push
```

推送后 GitHub Pages 约 1–2 分钟自动重新部署，访问 `https://<你的GitHub用户名>.github.io/<仓库名>` 即可查看。

### 首次启用 GitHub Pages（仅需一次）

若 Pages 尚未开启，告知用户：
1. 打开 `github.com/<用户名>/<仓库名>/settings/pages`
2. Source → 选 `Deploy from a branch`
3. Branch → 选 `main`，文件夹选 `/ (root)`
4. 点 Save，等待 1–2 分钟

### 版本管理日常命令

```bash
# 查看哪些文件有变动
git status

# 暂存简历相关文件
git add 简历/

# 提交（用公司+版本号描述，方便回溯）
git commit -m "add: [公司]_[方向] v1 (评审分数)"

# 推送
git push

# 查看历史版本
git log --oneline
```

## 建立你自己的"快速参考表"

每投几家公司后，把判断结论沉淀成一张表，后面遇到同类公司可以直接复用，不用每次重新分析：

| 公司/岗位 | 方向 | 特别说明 |
|---------|------|---------|
| [公司A] | 方向A | [可以直接复用的已有经历/案例] |
| [公司B] | 方向B | [这家公司最看重的关键词] |
| [公司C] | 混合 | [交叉定位怎么写的简短备注] |
| ... | ... | ... |

建表的方法：每次定制完一家公司后，把"判断结果 + 用了什么策略 + 投递反馈"记一行。投得越多，这张表越准，后面分析新公司时可以直接参考相似条目，省去重复推理的时间。
